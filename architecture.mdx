The application is a monolithic Next.js App Router project where every concern — portfolio, storefront, auth, payments, and administration — lives in the same codebase and shares the same database. This keeps deployment simple: one repository, one Vercel project, one MySQL database. The sections below describe how each layer is structured and how they connect.

## Request flow

A typical purchase looks like this:

<Steps>
  <Step title="Authentication check">
    The Next.js middleware (`middleware.ts`) runs on every request that is not a static asset or API call. It reads the JWT session and enforces two rules: unauthenticated users are redirected to `/auth/signin` if they try to reach `/admin`, and non-admin users are redirected to `/` if they reach any `/admin` route.
  </Step>
  <Step title="Product page & checkout">
    The customer browses `/shop`, selects a product, and clicks checkout. A Server Action (or API route) creates a Stripe Checkout Session or prepares a PayPal order. The customer completes payment on the provider's hosted page.
  </Step>
  <Step title="Webhook processing">
    Stripe sends a `checkout.session.completed` event to `/api/webhooks/stripe`. The handler verifies the signature with `STRIPE_WEBHOOK_SECRET`, marks the order as completed in MySQL, and generates license keys for each purchased product.
  </Step>
  <Step title="Post-purchase delivery">
    After checkout, the customer lands on `/shop/success`. License keys are accessible at `/licenses`. If the product has a `filePath`, the customer can download it through `/api/download` which validates the active license before serving the file.
  </Step>
</Steps>

## Data layer

The application uses Prisma ORM with a MySQL database. The schema (`prisma/schema.prisma`) defines the following core models:

| Model | Purpose |
|---|---|
| `User` | Registered users with role (`CUSTOMER` or `ADMIN`) |
| `Account` / `Session` | NextAuth.js OAuth account links and sessions |
| `Product` | Digital products with pricing, stock, and optional file path |
| `Category` | Product categories with slug-based routing |
| `Order` / `OrderItem` | Purchase records linked to Stripe sessions |
| `LicenseKey` | Generated keys tied to a user, product, and order |
| `Coupon` | Discount codes (percentage or fixed amount) |
| `PaymentMethod` | Enabled payment providers and their configuration |
| `SiteSettings` | JSON key-value store for admin-configurable settings |
| `Ticket` / `ChatMessage` | Support ticket system with live chat |
| `CarouselItem` | Homepage hero carousel content |

<Note>
  Prices are stored in cents (integers) throughout the schema, matching Stripe's convention. Always divide by 100 when displaying prices to users.
</Note>

## Authentication layer

NextAuth.js v5 is configured in `auth.config.ts` with two OAuth providers (GitHub and Google) and JWT sessions. The JWT callback copies the `role` field from the database user record into the token so that the middleware and server components can access the role without an extra database query.

```
Request
  └─ middleware.ts          reads JWT session, enforces role rules
       └─ auth.config.ts    JWT + session callbacks, provider config
            └─ NextAuth     issues/validates tokens, manages OAuth flow
                 └─ Prisma adapter  persists users, accounts, sessions to MySQL
```

## Payment flow

Two payment providers are supported and can be toggled independently from the admin dashboard via the `PaymentMethod` table:

```
Customer selects product
  ├─ Stripe path
  │    └─ /api/checkout        creates Stripe Checkout Session
  │         └─ Stripe hosted page
  │              └─ /api/webhooks/stripe   webhook: create Order + LicenseKey
  │
  └─ PayPal path
       └─ PayPal button (client-side @paypal/react-paypal-js)
            └─ /api/payment    server-side order capture + Order + LicenseKey
```
