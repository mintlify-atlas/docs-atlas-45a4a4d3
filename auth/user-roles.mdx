Every account on the Levi.dev portfolio has a role that determines what it can see and do. Most users are assigned the **CUSTOMER** role when they register. The **ADMIN** role unlocks full control over the site, including the admin dashboard. Two additional staff roles — **MODERATOR** and **SUPPORT** — are defined in the role system with permissions intended for partial admin access, though dashboard access is currently enforced at the middleware level for `ADMIN` only.

## Role overview

| Role | Default for new accounts | Admin dashboard access |
|------|:---:|:---:|
| CUSTOMER | Yes | No |
| SUPPORT | No | No (defined in role system) |
| MODERATOR | No | No (defined in role system) |
| ADMIN | No | Full |

## Staff roles

Two additional roles — MODERATOR and SUPPORT — are defined in the role system (`lib/roles.ts`) with permission sets for chat moderation and ticket management respectively. These roles are reserved for future use or custom access control logic added by the site owner.

| Role | Defined permissions |
|------|--------|
| **MODERATOR** | `view_admin`, `manage_chat`, `manage_tickets` |
| **SUPPORT** | `view_admin`, `manage_tickets` |

<Note>
  The middleware currently grants `/admin` access only to users with the `ADMIN` role. MODERATOR and SUPPORT roles are defined but do not grant dashboard access by default unless the middleware is extended.
</Note>

## How roles are enforced

Roles are stored in the database on your user record and included in your session token. Every time you load a page, the session is checked against the current value in the database so that role changes take effect immediately without requiring you to sign out.

The `/admin` routes are protected at the middleware level. If you are not signed in, visiting any `/admin` URL redirects you to the sign-in page. If you are signed in but your role is CUSTOMER, you are redirected to the home page instead.

<Note>
  Roles are not self-service. You cannot upgrade your own role from the account settings page.
</Note>

## How to become an admin

Admin access is granted directly in the database. The project includes two Prisma utility scripts for this purpose:

- `prisma/make-admin.js` — promotes a user to ADMIN by email address
- `prisma/promote-admin.js` — an alternative promotion script

These scripts are run by whoever manages the deployment environment. Contact the site owner if you need elevated access for a legitimate reason.
