---
title: 'License Keys: Generation, Statuses, Expiry, and Downloads'
description: 'How license keys are generated after purchase, the ACTIVE/REVOKED/EXPIRED status lifecycle, duration-based expiry, and the authenticated file download endpoint.'
---

Every completed purchase generates one `LicenseKey` record per product in the order. The license key is the mechanism that ties a customer to their purchased product — it gates file downloads, tracks access status, and carries expiry information when the product has a time-limited license.

## LicenseKey model

```prisma prisma/schema.prisma
model LicenseKey {
  id        String   @id @default(cuid())
  key       String   @unique
  productId String
  userId    String
  orderId   String
  status    String   @default("ACTIVE") // ACTIVE, REVOKED, USED, EXPIRED
  expiresAt DateTime?
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  product Product @relation(fields: [productId], references: [id])
  user    User    @relation(fields: [userId], references: [id], onDelete: Cascade)
  order   Order   @relation(fields: [orderId], references: [id], onDelete: Cascade)
}
```

### Field reference

| Field | Type | Notes |
|---|---|---|
| `key` | `String` (unique) | The formatted license string shown to the customer, e.g. `A1B2-C3D4-E5F6-G7H8` |
| `productId` | `String` | Foreign key to the purchased `Product` |
| `userId` | `String` | Foreign key to the `User` who made the purchase |
| `orderId` | `String` | Foreign key to the parent `Order` |
| `status` | `String` | Current key state — one of `ACTIVE`, `REVOKED`, `USED`, or `EXPIRED` |
| `expiresAt` | `DateTime?` | UTC expiry timestamp; `null` means the key never expires (lifetime license) |

## Key generation

License keys are generated by the `generateLicenseKey()` function in `lib/license.ts`. It creates a UUID v4, strips the hyphens, uppercases the result, and formats the first 16 hex characters into four groups of four separated by dashes:

```typescript lib/license.ts
import { v4 as uuidv4 } from 'uuid';

export function generateLicenseKey(): string {
  const uuid = uuidv4().replace(/-/g, '').toUpperCase();

  const part1 = uuid.substring(0, 4);
  const part2 = uuid.substring(4, 8);
  const part3 = uuid.substring(8, 12);
  const part4 = uuid.substring(12, 16);

  return `${part1}-${part2}-${part3}-${part4}`;
}
```

A generated key looks like: `A1B2-C3D4-E5F6-G7H8`

Keys are created inside the payment webhook (Stripe) or capture route (PayPal) — never before payment is confirmed. One key is created per product per order, even if the same customer purchases the same product twice.

## Expiry calculation

The `duration` field on the `Product` model determines how long a license key remains valid after purchase.

```typescript app/api/webhooks/stripe/route.ts
const expiresAt = product?.duration
  ? new Date(Date.now() + product.duration * 24 * 60 * 60 * 1000)
  : null

await db.licenseKey.create({
  data: {
    key: generateLicenseKey(),
    productId,
    userId: user.id,
    orderId: order.id,
    status: "ACTIVE",
    expiresAt,
  }
})
```

| `Product.duration` | `LicenseKey.expiresAt` |
|---|---|
| `null` | `null` — lifetime access, key never expires |
| `30` | `purchaseTimestamp + 30 days` |
| `90` | `purchaseTimestamp + 90 days` |
| `365` | `purchaseTimestamp + 365 days` |

<Note>
  Expiry is written at the moment the webhook fires. The database stores the absolute expiry timestamp, not the duration. If a product's `duration` field is changed after purchase, existing license keys are not affected.
</Note>

## Key statuses

| Status | Meaning |
|---|---|
| `ACTIVE` | The key is valid and the download endpoint accepts it |
| `REVOKED` | An admin has manually invalidated the key; downloads are blocked |
| `USED` | Reserved for single-use keys; download endpoint blocks further access |
| `EXPIRED` | The key has passed its `expiresAt` timestamp (set manually or checked in code) |

<Warning>
  The download endpoint currently checks `status !== "ACTIVE"` and rejects all non-active keys. The `EXPIRED` status is stored in the schema but expiry enforcement via `expiresAt` timestamp comparison is not implemented in the download route — only the `status` field value is evaluated.
</Warning>

## Viewing license keys

Customers access their license keys from the orders page at `/shop/orders`. The page queries all orders for the authenticated user, including the nested `licenseKeys` and their associated products:

```typescript app/shop/orders/page.tsx
const orders = await db.order.findMany({
  where: { userId: session.user.id },
  include: {
    items: { include: { product: true } },
    licenseKeys: { include: { product: true } }
  },
  orderBy: { createdAt: 'desc' }
})
```

For each order item where the product has a `filePath` set, a download button is rendered linking to `/api/download/[licenseId]`. If the license key has not yet been created (e.g. still processing), a disabled "Processing License..." button is shown instead.

## Download endpoint

`GET /api/download/[licenseId]` serves the product file associated with a license key. The route performs the following checks before streaming the file:

<Steps>
  <Step title="Authentication check">
    The request must come from an authenticated session. Unauthenticated requests receive a `401 Unauthorized` response.
  </Step>
  <Step title="License lookup">
    The license key is fetched by ID with its related `product` and `user` records included.
  </Step>
  <Step title="Ownership verification">
    The email on the authenticated session must match the email of the user linked to the license. This prevents one user from using another user's license ID to download files.
  </Step>
  <Step title="Status check">
    The license `status` must be `ACTIVE`. Any other status returns `403 License is not active`.
  </Step>
  <Step title="File existence check">
    The product must have a non-null `filePath`. If it is null, the endpoint returns `404 No file associated with this product`.
  </Step>
  <Step title="File streaming">
    The file is read from `{cwd}/storage/products/{filePath}` and returned with `Content-Disposition: attachment` and `Content-Type: application/octet-stream`. The original filename is reconstructed by splitting `filePath` on `-` and discarding the first two segments (timestamp and random suffix).
  </Step>
</Steps>

```typescript app/api/download/[licenseId]/route.ts
const filePath = join(process.cwd(), "storage", "products", license.product.filePath)
const fileBuffer = await readFile(filePath)

const parts = license.product.filePath.split('-')
const originalName = parts.slice(2).join('-')

return new NextResponse(fileBuffer, {
  headers: {
    "Content-Disposition": `attachment; filename="${originalName}"`,
    "Content-Type": "application/octet-stream",
  },
})
```

<Tip>
  Product files must be stored in the `storage/products/` directory at the project root with the naming convention `{timestamp}-{random}-{originalFilename}`. This is the format produced by the admin file upload handler, and it is the format the download route expects when reconstructing the original filename.
</Tip>
